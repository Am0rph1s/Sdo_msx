# Devlog: Nau DX — From Amstrad CPC to MSX1

Nau DX started life as a horizontal shoot-em-up for the Amstrad CPC, a machine with a linear screen buffer, hardware scrolling tricks, and a proper 3-channel sound chip. Porting it to the ZX Spectrum was already done — that version wrestled with the beeper speaker, a non-linear display, and the total absence of hardware sprites. The MSX1 sits somewhere in between. It has hardware sprites, but they are one bit per pixel with a single color each. It has a sound chip, but it shares the joystick port. And it has a Z80 running slightly slower than the CPC's, fed by a C compiler that is not known for generating tight code. Here is how the port came together.

The first thing any CPC-to-MSX port has to confront is the sprites. On the CPC, a single sprite can have four colors with wide Mode 0 pixels — a ship can be white, cyan, red, and yellow all at once, defined in a single bitmap. The MSX video chip gives you fifteen colors, but only one per sprite. You cannot have a white body and a red cockpit in the same sprite.

The workaround is to split every sprite into two layers drawn at the same position: a white silhouette layer with the overall shape, and a red detail layer with the cockpit, engines, and accents. The VDP composites them automatically, and the result looks like a single multi-colored sprite even though it is technically two. Every enemy in the game — basic, fast, heavy, diver, bomber — plus the boss and the player ship all use this two-layer approach. It costs twice the sprite slots, but with thirty-two hardware sprites available there is enough room even with bullets and explosions on screen.

The CPC original has a linear screen buffer and smooth 50 fps performance because most of the heavy lifting is in hand-written assembler. The MSX port uses SDCC, a C compiler for Z80 that is solid but not great at tight loops. A Z80 has no cache, no pipelining, and multiplying by anything other than a power of two is expensive. The enemy update loop was the hot spot — for every active enemy, the code reads and writes its position, speed, pattern, health, zigzag timer, and firing cooldown up to forty times per frame. Each access through an array index recalculates the memory address. Simply caching a pointer to the current enemy in a local variable at the top of the loop lets the compiler use register-based addressing through the HL register instead of recomputing the offset every time. It is a single line of C that saves measurable CPU time every frame.

Another quiet optimization was in the sprite update. The VDP sprite attribute table is 128 bytes — four bytes for each of the thirty-two possible sprite slots. The original code filled all 128 bytes and sent them to the VDP every single frame. In practice, only a handful of sprites are active at any moment. The new code tracks how many slots were actually used and only transfers that range over the bus. When no sprites are active — during menus and the title screen — zero bytes are transferred. Every byte saved on the VDP bus is a byte the CPU can use for game logic instead of waiting.

The sound chip on the MSX is a Yamaha YM2149, a close cousin of the CPC's AY-3-8912. Three tone channels, one noise channel, an envelope generator. That part of the port was straightforward. What was not straightforward was discovering that the joystick port is wired directly to the same chip. Two pins of the sound chip double as input pins for the joystick direction and fire buttons. A single register, number seven, controls whether those pins listen for input or drive output. Set that register wrong and the joystick goes completely dead.

This is not a theoretical problem. On the Sony HB-75P and other MSX1 machines with a real YM2149, two bits in register seven are write-only — you can set them, but they always read back as zero no matter what you wrote. The MSXgl sound library has convenience functions that enable and disable individual tone channels. Internally, those functions read register seven, flip a bit, and write it back. Every time a sound effect played, the read returned zeros for those two pins, and the write-back cleared the joystick input mode. The ship would fire continuously and refuse to move because it was reading garbage from a port set to output mode.

The fix was to stop reading register seven entirely. Every place that modifies the mixer now computes the full eight-bit value from scratch. It knows which tone channels are active, which noise channels are muted, and it always sets the joystick port to input mode. The intro music — a short looping tune that plays over the title screen — needed the same treatment. Its original register value put both I/O ports in output mode, a configuration that some older MSX models explicitly warn about because it can cause electrical conflicts with the keyboard matrix. Two single-byte changes later, the music sounds identical and the warning in openMSX is gone.

The game logic is frame-based. One update per frame. On a European MSX running at 50 Hz, that means fifty updates per second. On a Japanese or American machine running at 60 Hz, it means sixty. Everything moves twenty percent faster — enemies shoot sooner, bullets cross the screen quicker, and the difficulty curve is not the one that was designed. The fix is to detect the refresh rate at startup using the VDP status register and, on 60 Hz hardware, skip exactly one frame in every six. Five game updates, one pause, repeat forever. The result is the same effective speed on any MSX anywhere in the world.

The cartridge format gives you 32 kilobytes of ROM. That is the code, the sprite data, the font, the level layouts, the title screen, the sound effects — everything. By the time all five enemy sprite pairs, the three explosion frames, both boss layers, the player ship, the thruster flame, and shots for both sides are accounted for, plus the game loop, collision detection, HUD, wall scrolling, starfield, and menu screen, the final build has eight bytes free. Eight. Out of thirty-two thousand.

When we added the intro screen — a full-color 16-color image with background music — it was immediately clear that 32KB was not going to cut it. The intro image alone is twelve kilobytes of raw screen data. The solution was a mapper chip, the Konami SCC, which expands the cartridge to 64KB by splitting it into eight switchable 8KB segments. The intro occupies the first two segments. The game binary is split across the next four. When the intro finishes — either because the music ends or the player presses fire — a tiny assembly routine swaps all four game banks into view simultaneously and jumps to the game's entry point. The transition takes a handful of cycles. One frame you are looking at the title screen, the next you are in the game.

The hardest bugs were not the ones we expected. For several days the game would corrupt itself in bizarre and inconsistent ways. Wrong tiles would appear on the left wall during boss fights. Enemies would ignore screen boundaries. The game would slow down for no apparent reason. Every symptom pointed to a memory overflow — too many variables, too large structs, stack and heap colliding. We reverted optimizations, unpadded data structures, stripped out variables. The corruption moved around but never fully disappeared.

The actual cause was three completely separate VRAM bugs that all happened to manifest at roughly the same time. The boss health bar draws five filled segments across the right side of the screen. The screen is thirty-two columns wide, numbered zero through thirty-one. The original code started drawing the bar at column twenty-eight. Five segments from twenty-eight reaches column thirty-two. In the VDP name table, column thirty-two of any row is the same memory location as column zero of the next row — exactly where the left wall lives. Every time the health bar updated, its fifth segment was silently overwriting the wall tile one row below.

The second bug was simpler: some HUD cleanup strings were ten characters long but started too close to the right edge. The last few characters wrapped around and erased wall tiles on the next row. The third was the missing frame-rate limiter — on a 60 Hz test machine, everything was running too fast, which made existing race conditions between VDP writes visible. None of these bugs had anything to do with RAM. All three were visual artifacts that impersonated memory corruption so convincingly that we spent a full day debugging the wrong thing.

With those three fixes applied — shift the health bar one column left, shorten the cleanup strings, and lock the frame rate — the game ran perfectly. The struct padding we had removed turned out to be harmless. The pointer caching stayed. The VDP partial writes stayed. The joystick worked on real hardware. The intro played music without warnings. The walls stayed where they belonged.

The result is a 50 Hz shooter, five enemy types, multi-tier bosses, starfield scrolling, twenty-five levels, configurable controls, and an intro screen with music — all running on an unexpanded MSX1. The standalone game ROM fits in 32KB with eight bytes to spare. The version with the intro uses a Konami SCC mapper cartridge.

Porting to the MSX1 means learning to share. The sound chip and the joystick share the same chip, and you cannot read without risking a write. The CPU and the VDP share the same bus, and every byte you save on VRAM transfers is a byte the CPU can use for something else. The screen has hard edges, and crossing them silently corrupts your neighbor's tiles. It is a machine that rewards knowing exactly where your bytes are going, because it will not tell you when they arrive at the wrong address.
